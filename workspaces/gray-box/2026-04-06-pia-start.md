## B. 为什么先选 PIA

`PIA` 是当前最短路径，不是因为它“最强”，而是因为它同时满足 4 个条件：

1. 它明确属于灰盒/半白盒访问，不会把黑盒线和灰盒线混掉。
2. Research 内已有最完整的执行链，包含 `plan -> asset probe -> dry-run -> runtime probe -> smoke`。
3. 本地 `external/PIA` 已经带上游 DDPM 代码和 member split，不需要先补仓库骨架。
4. 它未来接入统一系统时最自然，因为当前 `local_api` 和 CLI 都已经是“按 method + job_type 调度”的结构，`PIA` 比 caption-free / text-to-image 论文更容易先对齐接口。

`SecMI` 很接近，但它当前更像“第二条灰盒 baseline”。它依赖 `flagfile + checkpoint` 的上游目录布局，进入真实资产阶段前不如 `PIA` 直接。

## D. 未来如何进入统一系统

### 1. 灰盒线独有输入

灰盒线不能只靠黑盒那套 `dataset/model/workspace`。至少还要显式表达下面这些 gray-box-only 输入：

| 字段 | 作用 |
| --- | --- |
| `access_profile` | 攻击者能读到什么，例如 `epsilon_t`、`score_t`、`latent_t`、`x0/x_t pair`、`cross_attention`、`controllable_initial_noise` |
| `feature_access` | 哪些中间特征真的可见，例如 partial checkpoint、noise prediction、intermediate denoise stats、decoder/prior、caption embedding |
| `member_split_root` | gray-box 多数方法默认依赖成员/非成员划分文件 |
| `model_dir` | 指向真实 checkpoint 目录，而不是只写模型名 |
| `flagfile_path` | `SecMI` 一类方法需要上游训练 flagfile 才能解析模型形状 |
| `partial_checkpoint` | 某些方法不要求完整训练目录，但要求能读部分权重或某类头部参数 |
| `noise_probe_config` | `attack_num`、`interval`、`t_sec`、`timesteps_observed`、`initial_noise_source` 等时间/噪声探针配置 |
| `intermediate_stats_root` | 需要缓存中间去噪统计、feature dump 或 trajectory summary 的方法必须有独立输入根目录 |
| `caption_mode` | `caption-known` / `caption-unknown` / `caption-surrogate`，对 text-to-image 灰盒线是核心前提 |
| `inversion_backend` | `Noise as a Probe` / `MoFit` 这类路线需要 inversion 或 surrogate encoder 后端 |

### 2. 与黑盒共有的最小字段

黑盒和灰盒至少应该共享下面这 5 个最小字段，先把任务状态收进统一索引：

```json
{
  "attack_family": "membership_inference",
  "method": "pia",
  "workspace": "workspaces/gray-box/runs/pia-runtime-smoke-20260406-cpu",
  "status": "ready",
  "evidence_level": "smoke"
}
```

说明：

- `attack_family`：统一到任务族，例如 `membership_inference`
- `method`：具体方法，如 `pia` / `secmi` / `recon`
- `workspace`：唯一输出根
- `status`：`blocked / ready / error / running`
- `evidence_level`：`plan / probe / dry-run / smoke / runtime-mainline / paper-mainline`

这 5 个字段可以直接作为更大 schema 的索引层，后续再附挂 `track / paper / assets / checks / metrics / cost`。

### 3. 如果接入 Local-API，最小 `job_type`

按当前 `local_api` 的命名方式，灰盒最小 job 不应该先拆成 `pia_job`、`secmi_job` 这种方法专属类型，而应该先落一个通用入口：

```json
{
  "job_type": "graybox_runtime_mainline",
  "workspace_name": "graybox-pia-cifar10-smoke",
  "method": "pia",
  "mode": "smoke",
  "device": "cpu",
  "repo_root": "D:\\Code\\DiffAudit\\Research\\external\\PIA",
  "dataset_name": "cifar10",
  "dataset_root": "D:\\datasets",
  "model_dir": "D:\\models\\pia\\cifar10",
  "member_split_root": "D:\\Code\\DiffAudit\\Research\\external\\PIA\\DDPM",
  "access_profile": [
    "epsilon_t"
  ],
  "attack_params": {
    "attacker_name": "PIA",
    "attack_num": 30,
    "interval": 10
  }
}
```

最小设计原则：

- `job_type` 只表达调度层能力，例如 `graybox_runtime_mainline`
- `method` 决定内部路由到 `pia / secmi / clid / future_sima`
- `mode` 区分 `probe / smoke / full`
- `attack_params` 收容 method-specific 参数，避免把 Local-API schema 绑死在单论文上

如果后续真的接到 Python `local_api.app`，推荐先只支持：

1. `mode=probe`
2. `mode=smoke`
3. `method in {pia, secmi}`

不要一开始就把 `MoFit`、`Noise as a Probe`、`CDI` 一起塞进同一个首版 job router。
