# ELSA Health Privacy Challenge Gate

> Date: 2026-05-15
> Status: biomedical synthetic-data MIA benchmark watch-plus / public starter example packet / challenge data gated by registration and DDA / no diffusion-data download / no GPU release

## Question

Does the CAMDA 2026 / ELSA Health Privacy Challenge provide a new measurable
DiffAudit mainline, especially because it includes `Noisy Diffusion` synthetic
datasets and red-team membership inference tasks?

This is an artifact gate only. It inspects public website metadata, the public
GitHub starter package, and public example ZIPs. It does not register for the
platform, accept a Data Download Agreement, download challenge datasets, or run
attacks against restricted data.

## Decision

`biomedical synthetic-data MIA benchmark watch-plus / public starter example
packet / challenge data gated by registration and DDA / no diffusion-data
download / no GPU release`.

This is stronger than a paper-only item because it has a concrete benchmark
interface, baseline MIA code, public example packets, and a locally verifiable
label/prediction metric contract. It still does not release a DiffAudit mainline
execution task because the actual Noisy Diffusion synthetic datasets and
challenge metadata require platform registration and a data agreement, and the
domain is biomedical gene-expression synthetic data rather than the current
image/latent-image admitted bundle.

Stop condition:

- Do not register, accept agreements, or download ELSA challenge data inside
  the current image/latent-image roadmap cycle.
- Do not treat the public example AUC as a research result against Noisy
  Diffusion. It is only a starter packet sanity check.
- Do not add biomedical/tabular product rows, Runtime schemas, or admitted
  bundle entries without an explicit consumer-boundary decision.
- Reopen only if DiffAudit explicitly opens a biomedical synthetic-data MIA
  lane or if public-safe Noisy Diffusion target/split/score artifacts become
  available without agreement-gated data.

## Reflection

The public starter packet is useful because it demonstrates a real
label/prediction/metric contract and confirms that the challenge is not just a
paper pointer. The correct next state is still watch-plus, not execution. The
current Research slots remain `active_gpu_question = none`,
`next_gpu_candidate = none`, and `CPU sidecar = none selected after ELSA Health
Privacy Challenge gate`.
