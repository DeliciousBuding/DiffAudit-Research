I treated your Round 2 note as an implementation memo for the **current admitted DDPM/CIFAR10 gray-box surface**: no gradients, low-FPR first, CPU-first 64→128→256→512, and defense evaluation with fixed-budget adaptive review rather than one-shot non-adaptive numbers.         

Two corrections matter before the details.

First, for **SimA**, the public DDPM implementation is more concrete than the prose: the repo’s `SimA` path queries the model on **clean `x0` with timestep `t`**, uses an **ℓ4-type score** (`sum |ε̂|^4`), and sweeps a small timestep grid; the same repo also includes `SimA_MC`, which actually noises `x0` to `x_t` and averages 30 Monte Carlo estimates. For a first reproduction, I would treat plain `SimA` as the anchor and `SimA_MC` as the ablation. ([arXiv][1])

Second, for the **frequency line**, the CIFAR-10 result in FCRE points the opposite way from the medical-image headline: on CIFAR-10, **FCRE(L2)** beats **FCRE(L2+SSIM)**, so your first low-res CIFAR packet should start with **band-pass L2**, not SSIM. ([arXiv][2])

## 1) SimA single-query score norm

### 1.1 Optimal timestep selection

**Paper-backed answer**

SimA defines the attack score as (A(x,t)=|\hat\epsilon_\theta(x,t)|_p), classifies a sample as a member if (A(x,t)\le \tau), and says (\tau) should be calibrated on held-out validation data. The paper explicitly says very early steps (t<10) are unstable, the useful regime is roughly **(t\in[10,300])**, and late steps **(t\in[300,1000])** lose signal as the diffused distribution approaches isotropic Gaussian. It also says the best (t) is **dataset- and noise-schedule-dependent**. The public DDPM code does not hard-code one universal (t^*); it evaluates a timestep grid determined by `interval` and `attack_num` and reports AUROC / ASR / TPR@1%FPR per timestep. ([arXiv][1])

**Recommended DDPM/CIFAR10 default**

For your surface, start with a **6-point scout** on a **128 member / 128 non-member** calibration split:
`{20, 40, 80, 120, 200, 300}`.

Choose (t^*) by:

1. **TPR@1%FPR**,
2. tie-break by **TPR@0.1%FPR**,
3. then **AUC**.

That is more aligned with your promotion logic than picking by raw AUROC. If you want a repo-faithful first packet before widening the range, use the code-like grid `0,20,40,60,80,100`, then expand only if the signal survives.    

**Needs empirical validation**

Yes. (t^*) is not universal.

---

### 1.2 Norm standardization method

**Paper-backed answer**

The SimA paper’s core score is the **raw norm** (A(x,t)=|\hat\epsilon_\theta(x,t)|_p). It does **not** define the main attack as (|\hat\epsilon|/\sigma_t), and the public DDPM code likewise uses the raw (\ell_4)-style magnitude. The appendix even shows that for members near (t\to 0), the predicted-noise magnitude already scales with (\sigma_t), so dividing by (\sigma_t) is not part of the published method and can wash out the very time dependence SimA exploits. The repo’s extra per-timestep rescaling to ([0,1]) is only an **evaluation normalization** for ROC computation, not an attack-time formula. ([arXiv][1])

**Recommended DDPM/CIFAR10 default**

Use this as the primary score:

[
A_t(x)=\sum_i |\hat\epsilon_\theta(x,t)_i|^4
]

Then, only for **fusion across methods or timesteps**, standardize **empirically on calibration non-members**, not by noise schedule:

* either rank / percentile,
* or z-score at fixed (t).

If you still want a noise-level ablation, use:

[
\sigma_t=\sqrt{1-\bar\alpha_t}, \quad \bar\alpha_t=\prod_{s=1}^t(1-\beta_s)
]

with your linear DDPM schedule (\beta_1=10^{-4}, \beta_T=0.02), but I would mark `A_t / σ_t` as an **ablation**, not the default. 

**Needs empirical validation**

Yes for any (\sigma_t)-based normalization. No for raw SimA.

---

### 1.3 Cosine feature and input jitter

**Paper-backed answer**

I did **not** find these as SimA-defined features. They are sensible extensions, not paper-specified components. What the SimA derivation does imply is that for members at small (t), (\hat\epsilon_\theta(x,t)) is approximately a **positive scalar multiple of (x)**, so a cosine with the input can have a member-favoring sign near early timesteps. But the paper does not report an ablation showing that cosine materially helps over the norm. ([arXiv][1])

**Recommended DDPM/CIFAR10 default**

For a repo-faithful SimA path:

* cosine feature:
  [
  c_t(x)=\frac{\langle \hat\epsilon_\theta(x,t), x\rangle}{|\hat\epsilon_\theta(x,t)|_2|x|_2}
  ]
* if you use the noisy-input ablation (`SimA_MC`-style), replace (x) with (x_t).

Use cosine only as an **extra fusion feature**, not as the primary attack.

For jitter, use a tiny Gaussian perturbation on normalized ([-1,1]) inputs:

* (\delta_k \sim \mathcal N(0,\sigma_j^2 I)),
* (\sigma_j \in {0.003, 0.005, 0.01}),
* (K=4) jittered copies.

Then define:
[
J_t(x)=\mathrm{Var}_k\left[\log(A_t(x+\delta_k)+10^{-8})\right]
]

That adds 4 extra forward passes, so I would only test it **after** plain SimA. Your hardware/budget profile does not justify starting with a multiplied query cost on an unsupported add-on.  

**Needs empirical validation**

Yes. Both are extensions.

### 1.4 Complementarity with PIA

**Paper-backed answer**

I did **not** find a paper that directly reports SimA–PIA correlation or a formal fusion study. What is solid is that the two methods use different statistics:

* **PIA** uses **two-query trajectory consistency**,
* **SimA** uses a **single-timestep predicted-noise magnitude**. ([arXiv][1]) 

The best supporting evidence for fusion is indirect: low-FPR calibration work shows that **raw target-model scores still matter**, calibrated scores help in the tail, and combining both kinds of evidence improves low-FPR behavior better than using either alone. ([arXiv][3])

**Recommended DDPM/CIFAR10 default**

Before fusing, compute on the **same 128/128 calibration split**:

* Spearman (\rho(\text{PIA}, \text{SimA})),
* overlap of the top-1%-tail sets.

My practical rule:

* if (|\rho|<0.85), or
* top-1% overlap (<70%),

then fusion is worth trying. Even when AUROCs are similar, low-FPR gain can still come from disagreement in the extreme tail.  

**Needs empirical validation**

Yes.

## 2) Mid-frequency residual implementation

### 2.1 DCT/FFT band selection

**Paper-backed answer**

The frequency-bias paper works in the **Fourier domain**, defines a **radius-based frequency split**, and recommends high-frequency-threshold radii in **([3,10])** with a suppression factor in **([0.0,0.3])**. FCRE, by contrast, does **not** define a radial annulus on CIFAR-10; it uses **8×8 patches**, computes a **Laplacian score on the original image**, and keeps **mid-frequency patches** between thresholds (L_{\min}) and (L_{\max}). Its ablation shows **15% / 85%** is strong, but also says the optimal thresholds vary by dataset. ([arXiv][4])

**Recommended DDPM/CIFAR10 default**

For your first CIFAR packet, I would choose **FFT**, not DCT, because the paper’s radial definition transfers more naturally.

Per RGB channel:

1. compute `fft2`,
2. `fftshift`,
3. define centered radius
   [
   r(u,v)=\sqrt{(u-16)^2+(v-16)^2}
   ]
   on 32×32,
4. test annuli:

   * `[3,10]`,
   * `[4,10]`,
   * `[4,12]`.

Use the **same mask for all samples**, picked on calibration data.

If you want an FCRE-nearer implementation, skip annuli entirely and do:

* 8×8 patches,
* Laplacian score on the original (x_0),
* retain patches in the **15th–85th percentile**,
* compare only those patches.

Do **not** convert to grayscale for CIFAR-10; run per-channel and sum or average the band scores. ([arXiv][2])

**Needs empirical validation**

Yes. No paper gives the exact best CIFAR-10 annulus.

---

### 2.2 1-step vs 5-step reconstruction trade-off

**Paper-backed answer**

FCRE uses a **partial reverse DDIM process** with **(T=1000)** and **(k=100)** sampling steps, and its fundamental error is the same-noise-level reconstruction error (| \tilde x_t - x_t |), not necessarily (x_0 - \hat x_0). It does **not** give a CIFAR-specific “1-step vs 5-step” ablation. That split is a practical adaptation, not a paper-defined protocol. ([arXiv][2])

**Recommended DDPM/CIFAR10 default**

Use the **same-noise-level residual**, not (x_0-\hat x_0), for the first packet.

* **1-step scout**:

  * choose one timestep (t^*),
  * generate (x_{t^*}),
  * do one deterministic reverse/reconstruct operation,
  * compute
    [
    r_{t^*}=\tilde x_{t^*}-x_{t^*}
    ]

* **5-step variant**:

  * evaluate 5 timesteps up to (t^*),
  * average the band-pass residual score across them.

For CIFAR-10, I would not optimize PSNR/SSIM as a target. Optimize **TPR@1%FPR**. The residual should stay in the model’s normalized pixel space, and you can apply FFT/DCT directly to that residual.  

**Needs empirical validation**

Yes.

---

### 2.3 Band-pass SSIM vs L2 choice

**Paper-backed answer**

FCRE’s proposed combined score is:

[
(1-\mathrm{SSIM}) + |\tilde x_t^F - x_t^F|_2
]

where the superscript (F) denotes the mid-frequency-masked patches. But on **CIFAR-10 specifically**, the paper’s table shows:

* **FCRE (L2)**: better than
* **FCRE (L2+SSIM)**

on both **AUC** and **TPR@1%FPR**. The L2+SSIM gain is a medical-image effect, not a CIFAR-10 one. ([arXiv][2])

**Recommended DDPM/CIFAR10 default**

Start with **band-pass L2**:
[
S_{\text{freq}}(x)=|M\odot \mathcal F(r_t)|_2
]

where (M) is your band mask and (\mathcal F) is FFT (or DCT if you choose that route).

Only add SSIM as a later ablation. If you do, compute it on:

* masked spatial patches, or
* inverse-transformed masked reconstructions,

not directly on raw frequency coefficients.

**Needs empirical validation**

L2: no special ablation needed for first packet.
SSIM: yes.

## 3) Tail-calibrated fusion

### 3.1 PIA + SimA + frequency fusion strategy

**Paper-backed answer**

The diffusion quantile-regression paper does **not** prescribe multi-signal fusion. What it does prescribe is **per-example thresholding**: learn the (\tau)-quantile of the score on public data and declare “IN” if the target score is below that learned threshold. It also offers a bagged ensemble of weak quantile models. Separately, LDC-MIA shows that **raw target-model membership score + calibrated score** used jointly as features improves low-FPR performance better than relying on calibration alone. ([ar5iv][5])

**Recommended DDPM/CIFAR10 default**

Use this exact fusion pipeline:

1. Orient all scores so **larger = more member-like**:

   * (z_{\text{PIA}}=-R_{\text{PIA}})
   * (z_{\text{SimA}}=-A_{\text{SimA}})
   * (z_{\text{freq}}=-S_{\text{freq}})

2. Convert each to a **one-sided non-member p-value** on held-out calibration non-members:
   [
   p_j(x)=\frac{1+\sum_{u\in U_{\text{cal}}}\mathbf 1[z_j(u)\ge z_j(x)]}{|U_{\text{cal}}|+1}
   ]

3. Transform to a z-score:
   [
   a_j(x)=\Phi^{-1}(1-p_j(x))
   ]

4. Fuse with **nonnegative logistic regression**:
   [
   f(x)=\sigma!\left(b+\sum_j w_j a_j(x)\right),\quad w_j\ge 0
   ]

5. If calibration data are too small, fallback to:
   [
   f(x)=\frac{1}{m}\sum_j a_j(x)
   ]

I would **not** start with Fisher or product rules here: your signals are dependent, and your calibration split is small.  

**Needs empirical validation**

Yes. But this is the cleanest honest path around the current TMIA-DM 512 gap.

---

### 3.2 Nonmember-calibrated p-value calculation

**Paper-backed answer**

The quantile-regression attack is one-sided and explicitly ties the threshold to a target FPR. Older high-precision calibration work also calibrates thresholds using **public non-members** at a chosen false-positive tolerance. ([ar5iv][5])

**Recommended DDPM/CIFAR10 default**

* Use a calibration split disjoint from the final 512/512 test set.
* Use **non-members only** for the p-value transform.
* Use **members + non-members** only when fitting the fusion weights.

Procedure:

* reserve **128 non-members** for p-value calibration,
* reserve **128 members + the same 128 non-members** for weight fitting.

Use the one-sided right tail **after** orienting the scores as above. Handle ties with `>=` and the `+1` smoothing shown in the formula.

Because your calibration set is small, I would **not** do class-conditional calibration as the default, even though class-specific thresholds can help in other MIA settings. ([arXiv][3])

**Needs empirical validation**

No for the mechanics. Yes for whether it beats a simpler rank-average.

---

### 3.3 Monotone low-FPR objective

**Paper-backed answer**

In this context, “monotone” should mean: if any component becomes more member-like, the fused score should not move in the opposite direction. The QR paper gives an FPR-controlled thresholding idea; LDC shows the low-FPR regime is exactly where careful combination matters. ([ar5iv][5])

**Recommended DDPM/CIFAR10 default**

Use:

* **nonnegative weights** (w_j\ge 0), or isotonic regression if you want a stricter monotonicity guarantee.

Optimize:

1. **bootstrapped mean TPR@1%FPR** on calibration,
2. tie-break by **TPR@0.1%FPR**,
3. then **AUC**.

With only 128 calibration non-members, 1% FPR is coarse, so use 128/128 only to **shortlist**. Freeze weights after a **256/256** re-check. This matches your pipeline much better than spending budget on a fancy optimizer. A small grid or nonnegative logistic fit is enough.   

**Needs empirical validation**

Yes.

## 4) Suspicion-gated defense

### 4.1 Cheap detector choice

**Paper-backed answer**

For your unconditional DDPM/CIFAR10 setting, the best paper-backed cheap detector is still **SimA**: one query, gray-box compliant, and already on your admitted surface. AMG supports the broader idea of **automatic per-step detection plus selective intervention**, and it is demonstrated on **DDPM/CIFAR10**, but it does not give you a ready-made SimA-vs-frequency detector comparison for your setup. ([arXiv][6])

**Recommended DDPM/CIFAR10 default**

Use:
[
d(x)=A(x,t_{\text{det}})
]

with (t_{\text{det}}) taken from the SimA sweep. Start with **40 or 80** if you want a fixed first try.

I would **not** force an “earlier” timestep than the best SimA one, because on your setup the cost of one forward pass is basically the same regardless of which (t) you choose. Use the same (t^*) unless calibration clearly says otherwise.

Use frequency residual as the backup detector only if plain SimA detector quality is weak.

**Needs empirical validation**

Yes. I did not find a paper-backed detector AUROC for your exact DDPM/CIFAR10 split.

---

### 4.2 Routing threshold (q) selection

**Paper-backed answer**

I did not find a universal paper-backed top-(q) rule for unconditional DDPM/CIFAR10. AMG gives step-gated activation, not a ready-made percentile routing schedule for your threat model. ([arXiv][6])

**Recommended DDPM/CIFAR10 default**

Use a **percentile threshold**, not an absolute score threshold:

* (q \in {0.05, 0.10, 0.20})

Start with **(q=0.10)**.

Choose (q) on held-out calibration by maximizing:
[
\frac{-\Delta \text{TPR@1%FPR}}{\Delta \text{runtime}}
]

or, more simply:

1. privacy gain first,
2. overhead second,
3. quality third.

That is the most natural way to stay near your current low-overhead stochastic-dropout baseline instead of creating a defense that only wins by becoming much more expensive. Keep the **attacker budget fixed** and include repeated-query evaluation.  

**Needs empirical validation**

Yes.

---

### 4.3 Late-step perturbation strategy

**Paper-backed answer**

AMG’s DDPM-specific update shows a **step-gated intervention** of the form
[
x_{t-1}\leftarrow \mathcal N(\mu - \mathbf 1_{{\sigma_t>\lambda_t}}\Sigma G_{\text{sim}}, \Sigma)
]
and the supplemental emphasizes schedule-shaped activation rather than uniform intervention. On your own side, the currently admitted defense already has `all_steps` and `late_steps_only` variants of stochastic dropout. ([CVF开放获取][7]) 

**Recommended DDPM/CIFAR10 default**

For the first gated-defense packet, do **not** invent a brand-new perturbation. Reuse the thing you already have.

Use this:

1. detector = SimA at (t_{\text{det}}),
2. if sample is in top-(q) suspicious percentile,
3. activate your existing **stochastic-dropout** only in the **final third of the reverse process**,
4. keep the current dropout rate unchanged.

That gives you a fair three-way comparison:

* no defense,
* current `all_steps`,
* suspicion-gated `late_steps_only`.

If you want a second perturbation family, use **epsilon-output noise** only for suspicious samples:
[
\hat\epsilon'=\hat\epsilon+\lambda,\mathrm{std}_{\text{cal}}(\hat\epsilon_t),\xi,\quad \xi\sim\mathcal N(0,I),\ \lambda\in{0.02,0.05,0.1}
]

I would leave **step skipping** for later; it is the highest-quality-risk option.

**Needs empirical validation**

Yes.

## 5) Cross-cutting implementation choices

### 5.1 Calibration set size and splitting

Use exactly this ladder:

* **64/64**: sign check and bug check only
* **128/128**: hyperparameter calibration
* **256/256**: validation / freeze
* **512/512**: final report

Keep the **same calibration split** for SimA, frequency, and fusion so the comparison is honest. This is already the rhythm your own notes want.     

For low-FPR stability:

* 128/128 is enough to shortlist,
* 256/256 is the first place I would freeze attack hyperparameters,
* 512/512 is the first place I would make claims,
* and TPR@0.1%FPR is still noisy at 512 because you only have about half a false positive in expectation at that operating point.   

### 5.2 Statistical significance testing

Follow your internal guide, with one small refinement.

Primary:

* **paired bootstrap**, 1000 resamples,
* recompute thresholds / fused weights inside each resample where applicable,
* report **95% CI** for all four metrics,
* especially for **TPR@1%FPR**. 

Secondary:

* paired t-test on per-resample metric differences,
* Bonferroni correction if you compare many variants. 

Promotion rule:

* keep your current **≥5% key-metric improvement** gate,
* but for low-FPR claims I would also require the **paired bootstrap difference to stay positive**.  

### 5.3 Computational budget allocation

Using your admitted PIA packet as the cost anchor:

* PIA on 512/512 takes about **213 s** and uses **60 forward passes per sample**,
* a 6-timestep SimA sweep is only **6 forward passes per sample**,
* so a SimA sweep is roughly **one tenth of PIA’s query cost** at the same batch regime.    ([GitHub][8])

That implies this budget order:

1. **SimA 64/64 scout**
2. **SimA 128/128 calibration**
3. **Frequency 1-step 64/64 then 128/128**
4. **Fusion on cached scores, CPU-only**
5. Promote only the best one to **256/256**
6. Then one **512/512** confirmation packet

For memory, SimA has essentially the same per-forward memory footprint as any other DDPM UNet query; the FFT/DCT cost on 32×32 is negligible. The only thing likely to bite you is repeated multi-step reconstruction, which is why I would keep the frequency line at **1-step first, 5-step second**.   

## The shortest honest first packet

If I were turning this into code on your current surface, I would do exactly this:

1. **Repo-faithful SimA**

   * clean-`x0` DDPM implementation,
   * (\ell_4)-style score,
   * timesteps `{20,40,80,120,200,300}`,
   * choose (t^*) by **TPR@1%FPR** on 128/128.

2. **PIA + SimA fusion**

   * non-member empirical p-values,
   * z-transform,
   * nonnegative logistic combiner,
   * re-check at 256/256 before 512/512.

3. **Suspicion-gated defense**

   * detector = SimA at (t^*),
   * (q=0.10),
   * activate existing stochastic dropout only in the final third of reverse steps,
   * evaluate with repeated-query adaptive review under fixed budget.    

That is the cleanest path that is still:

* inside your gray-box contract,
* aligned with your low-FPR priority,
* cheap on RTX 4070 8GB,
* and not blocked on TMIA-DM’s 512-sample gap.
