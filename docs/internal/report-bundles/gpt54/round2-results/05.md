结论先放前面：把 **H1 和 H2 绑成同一个 first packet**。一次 shared `GSA + PIA` score table extraction，同时跑 `best single / weighted average / 2-feature logistic / support-disconfirm-neutral`。`H4` 用同一张表做第二顺位；`H3` 只在 `H1/H2` 证明确有 low-FPR lift 后再做；`H5` 先停在 `logistic stacking`，不放 tiny MLP。原因很明确：当前 `I-C` 已经从 `agreement-first` 转成 **calibrated conditional evidence transfer**，而 repo 要修的是 **low-FPR tail**，不是再把全 ROC 漂亮一点。

## 1. Recommended execution order

| 顺位 | 假设                                       | 现在怎么做                                                                      | 是否 first implementation |
| -- | ---------------------------------------- | -------------------------------------------------------------------------- | ----------------------- |
| 1  | **H1: GSA + PIA calibrated late fusion** | 同一 shared pairboard 上跑 best single / weighted average / 2-feature logistic | **是**                   |
| 2  | **H2: support / disconfirm / neutral**   | 直接复用 H1 的 calibrated outputs，CPU-only                                      | **和 H1 同包**             |
| 3  | **H4: tail-gated cascade**               | 仍复用 H1 的 pairboard，只加 routing rule                                         | H1/H2 正增益后              |
| 4  | **H3: Recon + PIA**                      | 作为 portable track，但先看 H1/H2 是否真有 tail complementarity                      | 后做                      |
| 5  | **H5: stacking**                         | logistic first；tiny MLP 上锁                                                 | 最后                      |

这个顺序和 handoff 是一致的：`GSA` 是最强 anchor，`PIA` 是最自然辅助信号，`Recon + PIA` 更 portable 但更重，`agreement-first` 已经被 falsify，所以最值得先做的是 `GSA + PIA` 的 calibrated late fusion，再把 selective/neutral 规则接在同一张表上。

执行上我会把它拆成四个 packet，而不是一次做满：

* `64`：只做 schema smoke，确认 shared availability、score direction、cache artifact 没问题
* `128`：选择 calibrator 和 H1 的默认 rule
* `256`：做第一次 validation
* `512`：做 promote / kill 决策

这和 repo 的 `CPU-first / bounded review / 64→128→256→512` 治理是一致的；而且 H1/H2 的 model fitting 基本都在 CPU 上，GPU 只花在已有 score extraction 上。

## 2. Shared score table design

不要直接做“每个方法一个 CSV”。要有一张 **canonical raw score table**，再派生 fold-local calibrated table。最稳的 schema 是：

```text
ids:
  sample_id, member_label, class_id(optional), repeat_id, split_role(cal/test),
  access_level(black/gray/white), availability_mask

raw:
  raw_score_gsa, raw_score_pia, raw_score_recon,
  direction_aligned_score_*   # 统一成“越大越像 member”
  extract_ms_*, query_count_*, config_hash_*, artifact_hash_*

derived_per_split_only:
  pct_*            # calibration-set ECDF percentile
  rank_*           # calibration-set rank
  p_cal_*          # isotonic or Venn-Abers calibrated prob
  p_lo_*, p_hi_*   # Venn-Abers interval, if used
  tau_1pct_*       # 在 calibration side 冻住
  tau_0_1pct_*     # 同上
  fused_score
  fused_prob
  route_flag
  state ∈ {support, disconfirm, neutral}
  pred_binary
```

关键点有三个。第一，**raw 和 derived 必须分层**，不能把全局 rank、全局 calibrated prob 写回 raw table；那样最容易发生 split leakage。第二，**shared score table 只在共同可用样本上比较**，比如 `GSA + PIA` 用 pairwise shared rows，`Recon + PIA` 另起 shared subset，并显式报告 attrition。第三，baseline 一律是 **同一 shared split 上重算的 best single**，不是 repo 里各盒子自己的 headline number。

我还会加一个实现约束：**所有 percentile / rank transform 都必须只用 calibration split 的经验分布来拟合，再映射到 test split**。绝对不要在 full table 上先算 percentile 再切分。这个点很小，但会直接决定评估诚实度。

## 3. Calibration design

### 默认选择

* **默认先用 isotonic**
* **只有两种情况才升到 Venn-Abers**

  1. calibration split 明显偏小，isotonic 在阈值附近出现大台阶、很不稳
  2. H2 真的需要 interval-aware 决策，把 uncertainty 显式写进 `neutral`

这和当前 brief 一致：先做轻量、单调、CPU-friendly 的 calibration；`Venn-Abers` 是增强版，不是默认起点。

### raw score drift 怎么检测，何时启用 rank transform

我会把 `rank/percentile` 当成 **drift fallback**，不是默认替代 raw。触发条件用下面三条，满足两条就启用：

1. `tau_1pct` 的位置在不同 repeats 之间飘得很厉害

   * 例如从 score-space 看阈值波动很大，或者折算到 percentile 后仍波动 > 10 个百分点
2. raw-score calibration 后，外层 repeats 上的系数/权重频繁翻号或极不稳定
3. calibration/test 的 raw-score 分布出现明显 shift

   * 用 KS / Wasserstein / PSI 任一都行，重点不是统计学花样，而是记录“raw scale 在不同 split 上不共尺”

触发后做法不是“只保留 rank”，而是 **并行跑两版**：

* raw → calibrated prob
* percentile → calibrated prob

谁在 outer evaluation 上更稳，就留谁做默认。这样最符合当前“先 honest，再复杂”的治理。

### 阈值冻结

评估时必须是 **fold-local / repeat-local 冻结**：

1. 只用 calibration split 拟合 calibrator
2. 只用 calibration split 定 `1%FPR / 0.1%FPR` 阈值
3. 把这套 frozen mapping 应到 test split

最终真的要写成 repo 默认时，再用全量 admitted calibration data 重拟合一次并冻结。评估阶段绝不能用 global threshold。

## 4. Fusion rule design

### H1: weighted average vs 2-feature logistic

H1 不需要先押宝其一；同一张表上两者都跑。

**Weighted average**

```text
p_fuse = w * p_GSA + (1 - w) * p_PIA
w ∈ {0.00, 0.05, ..., 1.00}
```

`w` 的选择标准不是 AUC，而是 calibration side 上的：

1. `TPR@1%FPR`
2. tie-break: `TPR@0.1%FPR`
3. 再 tie-break: Brier / log-loss 更稳的那一个

**2-feature logistic**

```text
logit(p_fuse) = β0 + β1 * p_GSA + β2 * p_PIA
```

我的建议是 ridge-leaning elastic-net，而不是纯 lasso。因为特征数极小，不需要靠 lasso 做硬筛选。参数就用：

* `l1_ratio ∈ {0, 0.1, 0.3, 0.5}`
* `C ∈ {0.01, 0.1, 1, 10, 100}`

不要直接用 `LogisticRegressionCV` 的默认 scorer；它会偏 log-loss/AUC。这里应该手写 inner selection，目标函数就是 low-FPR。
`logistic` 只有在下面条件同时满足时才替代 weighted average：

* mean `ΔTPR@1%FPR` 相对 weighted average ≥ `+0.01` 绝对值
* 5 个 outer repeats 里至少 4 个为正
* `TPR@0.1%FPR` 不更差
* `β1, β2` 符号稳定，不出现大范围翻号

否则就把 weighted average 作为 H1 的默认 rule。这样能最大化“增益来自真实 complementarity，而不是参数自由度”。

### H2: support / disconfirm / neutral

我建议把三态定义得更保守一点：

```text
support(member):
  p_GSA >= τG+  and  p_PIA >= τP+  and  |p_GSA - p_PIA| <= ε

disconfirm(non-member):
  p_GSA <= τG-  and  p_PIA <= τP-

neutral:
  其余全部，尤其是一高一低的高冲突样本
```

这比“一个高一个低就直接判 non-member”更诚实，因为当前 cross-box 的目标是低 FPR 的正类判断，不是把 conflict 强行解释成负类。
如果用了 Venn-Abers，就把上面的点概率条件换成 interval 条件：

* `support`: 两个 lower bound 都高于正阈值
* `disconfirm`: 两个 upper bound 都低于负阈值
* `neutral`: 区间跨阈值或互相冲突

`abstention / coverage` 必须进两张表：

1. **strict binary board**：把 `neutral` 记作 negative，仍然在全 test set 上算 4 个主指标
2. **selective board**：只看 `non-neutral` 样本，另外报告 `coverage / abstention / disagreement`

这样就不会靠大量 abstain 去“伪造提升”。
`conflict discounting` 可以显式建模，但先只允许一个参数：

```text
p_disc = p_fuse - γ * |p_GSA - p_PIA|
γ ∈ {0, 0.1, 0.2, 0.3}
```

`γ` 只在 H2/H4 里用，不要一上来把它塞进所有模型。

### H4: tail-gated cascade

band 不要按 raw score interval 定，**按校准后的 margin / quantile 定**。默认规则：

```text
route if |p_GSA - τ_GSA,1pct| <= δ
```

其中 `δ` 不直接用固定数值，而是反推 routed fraction，先扫 `{10%, 20%, 30%}`。
被 route 的样本上，用 H1 或 H2；没被 route 的样本，直接沿用 GSA。这样 band 对 drift 更稳，也更符合 low-FPR threshold repair 的目标。
cost-benefit ratio 这样记最干净：

```text
relative_overhead
= (avg_cost_method - avg_cost_GSA_only) / avg_cost_GSA_only
```

如果用文档里的 timing，`PIA/GSA ≈ 8.5/34 ≈ 0.25`，那 routed `20%` 的额外成本大约是 `5%`；这正是 H4 值得排在 H3 前面的原因。

### H3: Recon + PIA 的先验互补性怎么判

先不要因为“黑灰盒更 portable”就直接做。先看三件事：

* `tail error Jaccard`：在 fixed `1%FPR` 下两者错集重合有多高
* `rescue rate`：`PIA` 对 `Recon` misses 的补救率
* `FP amplification`：辅助信号是否同时放大了 anchor 的假阳性

只有当“补救 misses 的收益”明显大于“放大 FPs 的代价”时，H3 才值得往前推。否则它只是更贵的 portability story。黑盒/灰盒的 threat-model 价值是有的，但不应先于 low-FPR truth。

### H5: logistic stacking 的最小特征集，以及 tiny MLP 何时才值得

最小特征集我建议只用 **4 个**：

```text
p_GSA, p_PIA, p_GSA * p_PIA, |p_GSA - p_PIA|
```

如果 raw-scale drift 明显，再加：

```text
rank_GSA, rank_PIA
```

也就是 **4-feature base / 6-feature drift-robust variant**。
`max/min` 是冗余的，`ratio` 在小概率区不稳定，先不要。
高阶多项式、平方项、`both_high` 指示项也都先不要；`interaction + disagreement` 已经足够表达“同向增强”和“冲突惩罚”。

tiny MLP 只有在下面条件都满足时才允许试：

* logistic 相对 best single 已经稳定为正
* 加 interaction 后仍然留下明显的系统性残差
* 训练样本量已经至少到 outer train 级别的稳定区
* MLP 相对 logistic 有 **`> +0.02` 绝对 `ΔTPR@1%FPR`**
* 5 个 outer repeats 里至少 4 个为正
* gain 的 std ≤ `0.015`
* 没有任何一个 repeat 比 logistic 差超过 `0.01`

架构就锁死在极小版本：

* `[8, 4]`
* `ReLU`
* `dropout = 0.1`
* 单 sigmoid 输出
* `weight_decay = 1e-4`
* early stopping 看 validation `log-loss / Brier`，patience `10–15`

也就是说，**MLP 是被证明值得才准入，不是默认比较对象**。

## 5. Evaluation protocol

这里我会直接改掉“天真 5-fold”的默认想法。对 low-FPR 来说，`1024` 总样本做普通 5-fold，每个 outer test 太小，`1%FPR` 和尤其 `0.1%FPR` 都会非常离散。更合理的是：

* **outer evaluation**：`5` 次 **stratified 50/50 repeated holdout**

  * 每次都用同一套 repeat seeds 跑所有方法
  * 如果有 `class_id`，做 `membership × class` 联合分层
* **inner tuning**：只在 calibration half 里做

  * H1/H2/H4 几乎可以只用简单 grid
  * H5 才需要 inner CV

这比 textbook 5-fold 更适合 low-FPR threshold freeze。严格要做 5-fold 也可以，但那时应以 **pooled out-of-fold predictions** 算主板，而不是逐 fold 生硬平均。这个判断是我基于 packet size 做的推断；精确离散程度还取决于每个 packet 里 non-member 的实际数量。

主表固定两张：

**A. strict binary board**

* `AUC`
* `ASR`
* `TPR@1%FPR`  ← 主指标
* `TPR@0.1%FPR` ← 次指标

**B. selective board**

* `coverage`
* `abstention rate`
* `selective TPR@1%FPR`
* `selective risk`
* `disagreement/conflict rate`
* `relative_overhead`

这里要强调：promotion 看的是 **same shared split 上相对 best single 的 low-FPR lift**，不是 headline AUC，也不是跨 threat-model 的直接拼接。

calibration quality 我会只保留 4 个检查项：

* `Brier`
* `log-loss`
* `ECE`（`10` 个等频 bins，只做描述）
* threshold-local calibration（如果实现了 `CCE` 就用；没实现就看 1%FPR 阈值附近的局部 reliability）

统计上：

* 主结论用 **paired bootstrap 1000 次**，报 `ΔTPR@1%FPR` 的 `95% CI`
* 同时报 repeats 的 `mean ± std`
* split-level 的 paired `t-test` 只作辅证
* 多重比较用 `Holm-Bonferroni`；如果要完全贴当前 guide，用 plain `Bonferroni` 也行

缓存策略则反过来要“宽松但不泄漏”：

* **raw scores 可以一次性全量提取并缓存**
* calibration / threshold / percentile mapping **必须逐 repeat 单独拟合**
* 不要为了“避免泄漏”去重复做 GPU extraction；泄漏不在 raw score extraction，而在 label-aware 的 calibration stage

这完全符合当前 `CPU-first`：先在 cached scores 上做 CPU calibration / fusion，只有信号过了 `128→256→512` 的关口，才值得把它升成 admitted candidate。

## 6. Kill gates

我会把 kill 分成两层。

### 直接 kill

满足任一条，就认定当前 cross-box 不值得继续主推：

1. 在 `128` 或 `256` packet 上，best H1/H2 相对 same-split best single 的 `ΔTPR@1%FPR` 不是稳定正值
2. 提升只发生在 `AUC`，但 `TPR@1%FPR` 没提升或变差
3. logistic 的权重/符号在 repeats 之间明显不稳
4. `neutral` 太多，coverage 塌到 `< 60%` 还没有给 strict board 带来真实增益
5. H4 的额外 cost `> 20%`，却拿不到 `+0.02` 绝对 `TPR@1%FPR` 提升
6. H3 的 complementarity 检查表明 “救回 misses” 小于 “放大 FPs”
7. logistic 都没稳过 weighted average，却还想上 MLP

### 只保留为 bounded auxiliary evidence

这类不算主线失败，但也不该 promote：

* H2 只改善了 selective honesty，没有改善 strict binary board
* H4 只能当 cost-saver，不能当 performance booster
* 早期 packet 有小正增益，但 `256 → 512` 后消失
* gain 有，但没达到当前 promotion rule 的 **≥5% key-metric improvement** 或者 CI 明显跨 0

这和 repo 现有治理是一致的：**promote 要看 low-FPR、稳定性、成本；不满足就停在 bounded evidence，不要靠复杂模型硬续命。**

收束成一句实施命令，就是：

**先做一张 `GSA + PIA` shared pairboard；在这张表上完成 per-signal calibration、best-single / weighted / logistic / H2 三态；用 repeated 50/50 holdout 做 low-FPR board；只有这一步给出稳定 tail lift，才允许去做 H4，再之后才轮到 H3；H5 的 tiny MLP 继续上锁。**
