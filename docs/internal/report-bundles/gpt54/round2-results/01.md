## H2: Strength-Response Curve

Implement this one **paper-faithfully first**. *GenAI Confessions* uses image-to-image generation at multiple strengths, repeats generation at each strength, computes a perceptual distance from seed to output, keeps the **minimum distance per strength**, and feeds the resulting strength-vector into a **logistic regression** classifier. It also finds that out-of-training seeds become more distinct from the seed image as strength grows, and that this gap is informative. For DDPM/CIFAR10, I would mirror that structure with **4 moderate timesteps/strengths, 2 repeats each**, and use a cheap metric first (SSIM or L2; LPIPS only if the GPU budget allows). I would **not** start with a fancy curve model; first reproduce the vector-of-min-distances baseline, then add slope/AUC/curvature features as an ablation. The moderate-band choice is supported by REDIFFUSE, which reports that attack quality stays strong for **moderate diffusion steps** and falls off mainly when the step is too large. ([arXiv][1])

My concrete DDPM/CIFAR10 settings for H2 would be: a **64+64 scout**, four start points in the moderate band, two repeats each, min-distance retention per point, logistic regression on the 4-D vector, and bootstrap CIs on all four metrics. That fits the repo’s 64/128→256 progression much better than the earlier 256+256 all-at-once idea.   

## H1: Response-Cloud Geometry

Use **the same cached outputs as H2**, but score the **output cloud** instead of seed-to-output distance. REDIFFUSE supports the basic premise that repeated outputs from members cluster more tightly, because averaging multiple independent samples improves the attack; *GenAI Confessions* supports the same direction by showing less variation for in-training seeds under strength sweeps. So the H1 pilot should be a pure CPU-side feature pass over H2’s cached outputs: mean pairwise SSIM/L2, DCT-space covariance trace, top-eigenvalue share, and simple within-bag entropy. I would **not** add KDE / LiRA-style density fitting in the pilot; your own notes on distribution-based methods already warn that shadow-calibrated density layers are expensive and better treated as calibration, not the first signal family. ([arXiv][2]) 

The important design choice here is that H1 should stay **strictly output-output** in sprint 1. Once you reintroduce original-to-output distance too early, it starts collapsing back toward recon/variation. So I would treat H1 as “reuse H2 outputs, change only the scorer.” That keeps the GPU cost almost flat while giving you a genuinely different signal family. 

## H3: Frequency-Sliced Stability

The new files push H3 out of the “standalone sprint-1 candidate” role. The strongest paper support for H3 says frequency filtering is a **plug-and-play high-frequency filter** for existing error-based attacks: it removes confounding high-frequency variability, is implemented in the Fourier domain with a **radius-threshold mask**, and adds **negligible overhead**. That is a strong argument for using H3 as an **ablation on H2/H1**, not as a separate first-line scorer. Your own higher-level repo note also currently treats the frequency-domain paper as **explanation only**, which points the same way. ([arXiv][3]) 

So my implementation choice for H3 is: add a radial Fourier mask **before** scoring, sweep 3 candidate cutoffs on the 64+64 scout, and measure whether the filter improves **TPR@1%FPR** or reduces variance of the H2/H1 score. I would not claim a new standalone black-box family unless that filtered version later survives 256/512. On 32×32 CIFAR10, the cutoff should be treated as a calibration hyperparameter, not fixed from the text-to-image papers. ([arXiv][3]) 

## H5: Micro-Bag Statistical Audit

H5 is still attractive, but the new docs make its role clearer: it is best as a **statistical wrapper for weak signals**, not as the first pointwise attack you try to beat recon with. CDI explicitly argues that **single-image MIAs are often too weak** on large diffusion models, and gets its strength from **selective aggregation + a scoring model + formal statistical testing**, reaching >99% confidence with as few as 70 points. That means the literature-faithful version of H5 is: use it to stabilize low-FPR behavior or move toward set-level evidence, not to replace H2/H1 in the first sprint. ([CVF开放获取][4])

Implementation-wise, I would build H5 only after H2 or H1 gives you a usable scalar score. Then make an **8-variant bag** from the same image, standardize the per-variant scores, and aggregate with simple summaries plus a lightweight test. If that works, the honest promotion path is toward **CDI-style set-level claims**, not overclaiming a new single-image headline. That also aligns with your own “low-FPR matters” and statistical-testing notes.  

## H4: Counterfactual Patch-Recovery

This one should stay last. The best direct support I found is structural memorization: members’ **structures are better preserved** than non-members’, which is compatible with the counterfactual idea. But neither your repo notes nor the literature give a clean DDPM/CIFAR10 black-box recipe for patch-recovery, and your own deep-analysis already flags H4 as high-risk and under-specified. ([arXiv][5]) 

If you still want a bounded exploratory version, I would shrink the masks for CIFAR10 to **4×4 and 8×8 only**. I would not use 16×16 on 32×32 images in the first pass because that is too destructive and will mostly test generic completion rather than memorization. But I would treat any apparent win here as exploratory until it survives 256/512 with the full four-metric report.
